# Design: pgmq 1.11.0 Upgrade

## Overview

pgmq 1.11.0 introduces AMQP-like topic-based routing as the major new feature, along with batch topic sending, topic binding/throttle management functions, and several bug fixes. This document covers the changes required in pgmq-hs to support the upgrade.

Reference: `/docs/pgmq-1.10.0-to-1.11.0-changes.md` in the pgmq repository.

## Upstream Changes Summary

| Change | Category | pgmq-hs Impact |
|--------|----------|-----------------|
| AMQP-like topic routing | New feature | New types, statements, sessions, effectful |
| Batch topic sending | New feature | New types, statements, sessions, effectful |
| `list_topic_bindings()` | New function | New statement, session, effectful |
| `list_notify_insert_throttles()` | New function | New statement, session, effectful |
| `update_notify_insert()` | New function | New statement, session, effectful |
| `read_grouped` performance fix | Bug fix | No client changes needed (server-side) |
| `create_queue` missing `last_read_at` | Bug fix | No client changes needed (server-side) |
| Explicit `RETURNING` clauses | Bug fix | No client changes needed (server-side) |
| `send_batch` header validation | Behavioral change | No code changes, but existing mismatched calls will now raise errors |

## Upstream API

### Topic Binding Functions

```sql
-- Bind a wildcard pattern to a queue (idempotent)
pgmq.bind_topic(pattern TEXT, queue_name TEXT) -> VOID

-- Remove a binding; returns true if removed
pgmq.unbind_topic(pattern TEXT, queue_name TEXT) -> BOOLEAN

-- Validate a routing key (no wildcards, max 255 chars)
pgmq.validate_routing_key(routing_key TEXT) -> BOOLEAN

-- Validate a topic pattern (wildcards allowed)
pgmq.validate_topic_pattern(pattern TEXT) -> BOOLEAN

-- Dry-run: show which patterns/queues match a routing key
pgmq.test_routing(routing_key TEXT) -> TABLE(pattern TEXT, queue_name TEXT, compiled_regex TEXT)

-- List all topic bindings
pgmq.list_topic_bindings() -> TABLE(pattern TEXT, queue_name TEXT, bound_at TIMESTAMPTZ, compiled_regex TEXT)

-- List bindings for a specific queue
pgmq.list_topic_bindings(queue_name TEXT) -> TABLE(pattern TEXT, queue_name TEXT, bound_at TIMESTAMPTZ, compiled_regex TEXT)
```

### Topic Send Functions

```sql
-- Send to all queues matching routing key; returns count of matched queues
pgmq.send_topic(routing_key TEXT, msg JSONB) -> INTEGER
pgmq.send_topic(routing_key TEXT, msg JSONB, delay INTEGER) -> INTEGER
pgmq.send_topic(routing_key TEXT, msg JSONB, headers JSONB, delay INTEGER) -> INTEGER

-- Batch send to all matching queues
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[]) -> TABLE(queue_name TEXT, msg_id BIGINT)
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[], headers JSONB[]) -> TABLE(queue_name TEXT, msg_id BIGINT)
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[], delay INTEGER) -> TABLE(queue_name TEXT, msg_id BIGINT)
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[], delay TIMESTAMPTZ) -> TABLE(queue_name TEXT, msg_id BIGINT)
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[], headers JSONB[], delay INTEGER) -> TABLE(queue_name TEXT, msg_id BIGINT)
pgmq.send_batch_topic(routing_key TEXT, msgs JSONB[], headers JSONB[], delay TIMESTAMPTZ) -> TABLE(queue_name TEXT, msg_id BIGINT)
```

### Throttle Management Functions

```sql
-- List all notification insert throttle settings
pgmq.list_notify_insert_throttles() -> TABLE(queue_name TEXT, throttle_interval_ms INTEGER, last_notified_at TIMESTAMPTZ)

-- Update throttle interval for a queue
pgmq.update_notify_insert(queue_name TEXT, throttle_interval_ms INTEGER) -> VOID
```

## Proposed Changes

### 1. Core Types (`pgmq-core/src/Pgmq/Types.hs`)

Add a new `RoutingKey` validated newtype, similar to `QueueName`:

```haskell
-- pgmq-core/src/Pgmq/Types.hs

-- | A validated routing key for topic-based message routing (pgmq 1.11.0+)
-- Routing keys are dot-separated segments of alphanumeric characters, hyphens,
-- and underscores. Max 255 characters. No wildcards allowed.
newtype RoutingKey = RoutingKey Text
  deriving newtype (Eq, Ord, FromJSON, ToJSON)
  deriving stock (Show, Generic)

parseRoutingKey :: Text -> Either PgmqError RoutingKey
parseRoutingKey t
  | T.null t = Left $ InvalidRoutingKey "Routing key cannot be empty."
  | T.length t > 255 = Left $ InvalidRoutingKey "Routing key exceeds 255 characters."
  | not (T.all isValidChar t) = Left $ InvalidRoutingKey "Routing key contains invalid characters."
  | otherwise = Right $ RoutingKey t
  where
    isValidChar c = (isAscii c && isAlphaNum c) || c == '.' || c == '-' || c == '_'

routingKeyToText :: RoutingKey -> Text
routingKeyToText (RoutingKey t) = t

-- | A topic pattern for binding to queues (pgmq 1.11.0+)
-- Patterns support wildcards: '*' matches one segment, '#' matches zero or more.
newtype TopicPattern = TopicPattern Text
  deriving newtype (Eq, Ord, FromJSON, ToJSON)
  deriving stock (Show, Generic)

parseTopicPattern :: Text -> Either PgmqError TopicPattern
parseTopicPattern t
  | T.null t = Left $ InvalidTopicPattern "Topic pattern cannot be empty."
  | T.length t > 255 = Left $ InvalidTopicPattern "Topic pattern exceeds 255 characters."
  | otherwise = Right $ TopicPattern t

topicPatternToText :: TopicPattern -> Text
topicPatternToText (TopicPattern t) = t
```

Update `PgmqError` to include new error constructors:

```haskell
data PgmqError
  = InvalidQueueName Text
  | InvalidRoutingKey Text
  | InvalidTopicPattern Text
  deriving stock (Show, Generic)
```

Add a result type for topic binding info and topic send results:

```haskell
-- | A topic binding record returned by list_topic_bindings (pgmq 1.11.0+)

    queueName :: !Text,

### 8. Migration (`pgmq-migration`)

7. Fix `read_grouped` performance

- `pgmq--1.10.0--1.10.1.sql`

## Alternatives Considered

### Topic Type Design

**Option A: Validated newtypes for RoutingKey and TopicPattern (recommended)**
- **Pros:** Type safety prevents mixing routing keys and patterns; mirrors upstream validation rules; consistent with `QueueName` pattern
- **Cons:** More types to manage; users must construct via `parseRoutingKey`/`parseTopicPattern`

**Option B: Plain Text for both routing keys and patterns**
- **Pros:** Simpler API; server-side validation catches errors anyway
- **Cons:** No compile-time distinction between routing keys (no wildcards) and patterns (wildcards allowed); inconsistent with the `QueueName` pattern established in the codebase

**Option C: Single `TopicKey` type with a tag**
- **Pros:** One type to manage
- **Cons:** Overly complex; routing keys and patterns have different validation rules

### Topic Send Return Types

**Option A: Dedicated return types (recommended)**
- `sendTopic` returns `Int32` (matched queue count)
- `batchSendTopic` returns `[TopicSendResult]` with `queueName` + `messageId`
- **Pros:** Faithfully represents the upstream API; users know which queues received messages
- **Cons:** New types to define

**Option B: Reuse MessageId**
- **Pros:** Simpler
- **Cons:** Loses queue name information from batch results; misrepresents `sendTopic` which returns a count, not a message ID

### Module Organization for Topic Statements

**Option A: New `TopicManagement.hs` module (recommended)**
- **Pros:** Clean separation; topic routing is a distinct feature area; follows precedent of `QueueManagement.hs` vs `Message.hs`
- **Cons:** New file

**Option B: Add to existing `QueueManagement.hs`**
- **Pros:** No new file
- **Cons:** `QueueManagement.hs` grows large; topic operations are conceptually different from queue CRUD

**Option C: Add to existing `Message.hs`**
- **Pros:** Send functions are alongside other send functions
- **Cons:** Binding/routing functions don't belong in message operations; mixes concerns

## Implementation Notes

### Server-Side Bug Fixes (No Client Changes)

The following upstream fixes are transparent to pgmq-hs since we call the SQL functions by name and the return types haven't changed:

1. **`read_grouped` performance fix** — Internal query restructuring; same function signature and return type
2. **`create_queue` missing `last_read_at`** — DDL fix in queue creation functions; doesn't affect the client
3. **Explicit `RETURNING` clauses** — Ensures column ordering stability; pgmq-hs already decodes by position so this is a correctness improvement

### `send_batch` Validation Change

pgmq 1.11.0 now validates that `headers` array length matches `msgs` array length in `send_batch`. This could cause previously-silent mismatches to raise errors. No code changes needed in pgmq-hs, but this is a behavioral change worth noting in release notes.

### ForLater Variants for Topic Sends

The upstream `send_batch_topic` has overloads accepting `TIMESTAMP WITH TIME ZONE` for the delay parameter (schedule for a specific time). Following the existing pattern in pgmq-hs where `SendMessage`/`SendMessageForLater` and `BatchSendMessage`/`BatchSendMessageForLater` are separate types, we should provide `BatchSendTopicForLater` and `BatchSendTopicWithHeadersForLater` types.

For `sendTopic` (single message), the upstream API only has an `INTEGER` delay parameter (no timestamp overload), so we only need the delay-based variants.

bindTopic $ BindTopic
  { pattern = either (error . show) id $ parseTopicPattern "logs.#"
  , queueName = "all_logs"
  }

bindTopic $ BindTopic
  { pattern = either (error . show) id $ parseTopicPattern "*.error"
  , queueName = "error_logs"
  }

-- Send via topic routing (routes to all matching queues)
matchedCount <- sendTopic $ SendTopic
  { routingKey = either (error . show) id $ parseRoutingKey "logs.error"
  , messageBody = MessageBody $ object ["msg" .= ("something broke" :: Text)]
  , delay = Nothing
  }
-- matchedCount == 2 (all_logs + error_logs)

-- Dry-run: check which queues would match
matches <- testRouting $ either (error . show) id $ parseRoutingKey "logs.error"
-- matches contains RoutingMatch entries for "logs.#" -> all_logs and "*.error" -> error_logs

-- List all bindings
bindings <- listTopicBindings
-- bindings contains TopicBinding entries

-- Manage notification throttles
throttles <- listNotifyInsertThrottles
updateNotifyInsert $ UpdateNotifyInsert
  { queueName = "my_queue"
  , throttleIntervalMs = 500
  }
```

## Version Requirements

- All topic functions: pgmq 1.11.0+
- `listNotifyInsertThrottles`, `updateNotifyInsert`: pgmq 1.11.0+
